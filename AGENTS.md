# AGENTS.md

## 使用场景

本项目由产品经理本地使用，用于写方案、整理需求、做高保真原型和验证体验。  
除非用户明确要求，不面向生产，不主动引入复杂架构、后端、数据库、登录或部署流程。

## 核心原则

1. 默认使用中文，先给结论/方案，再补必要解释。
2. 少复述背景，少空话，避免把简单问题复杂化。
3. 普通细节可合理假设；影响方向、安全、数据、用户预期或实现范围的不确定点，先澄清再动手。
4. 不假装完成。没有运行/验证/截图，就明确说明“未运行”或“未验证”。

## 基础工作流

1. 先判断任务级别  
   - 小改动：文案、样式、布局、交互细节、配置修正，可直接执行。  
   - 中等改动：少量多文件、低风险逻辑调整，先简短列计划再执行。  
   - 大改动：涉及多页面、多流程、目录结构、核心状态、依赖安装、外部接口或数据结构，先规划再执行。

2. 动手前尽量拆成四段  
   `files`：影响哪些文件；`action`：做什么；`verify`：如何验证；`done`：完成后应看到什么。

3. 高风险操作先确认  
   删除文件、批量覆盖、安装依赖、改系统配置、运行不可逆命令、创建 branch / worktree 前，必须先说明风险并等确认。
余风险和建议验证方式。

4. 改完必须验证  
   优先本地运行、构建、类型检查或手动交互验证。若无法验证，说明原因、剩
## Skill / 工具使用

- 仅使用当前环境实际可用的 skill/tool/agent。
- 可提升质量的工具可优先使用；不可用时按工作流手工推进。
- 复杂任务可用 `<files>/<action>/<verify>/<done>` 维护计划。
- 实际使用了 skill/tool，且影响处理方式或用户需要复现时，在回复中说明使用情况和原因。

## 原型实现偏好

- 优先可看懂、可点击、可评审的实现。
- 保证体验闭环：页面访问、流程走通、交互反馈、状态清晰。
- 后台页面优先保证结构清晰，无异常换行、截断、遮挡、多余模块。
- 保持代码结构简单、命名清楚，便于理解和修改。
- 可使用 mock 数据，但不能掩盖真实依赖问题。

## 设计参考使用

- 有 Figma、截图或视觉参考时，默认按参考实现，不仅凭现有代码自行发挥。
- 涉及 Figma 时，先读结构、截图、变量、组件和 Library 信息；页面过大就拆 frame / section 读。
- 团队 Library / 设计系统可用时优先复用；不可用或未命中时明确说明，不伪装复用成功。
- 纯 HTML 原型最终交付 HTML/CSS/少量原生 JS；Figma MCP 的 React/Tailwind 只作中间参考。
- 实现前确认模块层级、尺寸间距、颜色字体、按钮/表单/表格/弹窗和关键文案。
- UI 任务完成后必须做浏览器级视觉自检；构建通过或无报错不等于视觉验收。
- 不用隐藏、裁切、绝对定位补丁、多层 CSS 覆盖等方式掩盖问题；无法还原就说明差异和原因。

## 输出规范

1. 复杂信息优先表格化  
   涉及判断、结论、对比、步骤、风险、文件变更、验证结果时，优先用 Markdown 表格呈现。

2. 简单问题短答；复杂问题按“判断 -> 方案 -> 步骤 -> 风险/验证”组织。

3. 用户要方案时，给清晰结构和可复制内容；用户要实现时，说明改了什么、怎么运行、如何验证。

4. 不捏造测试、运行结果或数据来源。没有实际运行就说“未运行”。
